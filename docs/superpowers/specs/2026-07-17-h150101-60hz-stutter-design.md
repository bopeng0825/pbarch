# H150101 60 Hz Audio/Video Stutter Design

## Goal

Keep NTSC PicoDrive content running at its original 60 Hz rate on H150101 while
prioritizing continuous audio and reducing visible unevenness when the device
cannot render every frame within the 16.67 ms budget.

## Evidence

Profiling shows that audio-buffer write blocking has been removed, but demanding
scenes still exceed the 60 Hz budget:

- PicoDrive core work (`inner`) varies from about 6 to 12 ms.
- RGB565 texture update costs about 7 to 8 ms per rendered frame.
- Flip and pacing usually cost another 1 to 2 ms.
- At 47 to 50 core runs per second, PicoDrive produces only 34.5 to 36.8 kHz
  worth of audio frames while the device consumes 44.1 kHz. Dynamic rate
  adjustment is too small to compensate for this sustained deficit.

## Selected Approach

Use a conservative combination of automatic frame skipping and removal of
unnecessary per-frame RGB565 copying.

### PicoDrive frame-skip policy

On H150101 builds only:

- Default `picodrive_frameskip` to `auto`.
- Default `picodrive_frameskip_interval` to `1`.
- Preserve user configuration: an explicitly saved value continues to override
  the platform default.
- Do not change defaults on other platforms.

Automatic frame skipping keeps emulation and audio generation at 60 Hz while
omitting a video render when the frontend reports that the audio buffer is at
risk. Limiting consecutive skips to one avoids long runs of missing frames.

### RGB565 upload path

For SDL2 hardware scaling:

- Upload the core's tightly packed RGB565 frame directly when no HUD message
  needs to be composited.
- Allocate and copy into the temporary RGB565 buffer only when drawing a HUD
  message.
- When entering the menu from hardware-scaled gameplay, capture the current
  renderer output once instead of maintaining a full shadow copy every frame.
- Keep the existing software-scaling and non-SDL2 paths unchanged.

This removes a redundant 320 x 224 x 2-byte copy from the normal gameplay hot
path while retaining menu-background and HUD behavior.

### Audio diagnostics

Extend profiling with:

- Audio callback underrun count.
- Minimum ring-buffer occupancy observed during the profile interval.

Reset both values with the existing one-second profile counters. These metrics
must not add logging outside the existing profile cadence.

## Error Handling

- If renderer readback fails during menu entry, clear the menu background and
  continue; menu access must remain functional.
- If the temporary HUD buffer cannot be allocated, fall back to uploading the
  unmodified core frame and retain the existing warning.
- Frame skipping remains controlled by PicoDrive and the existing libretro
  audio-buffer-status callback.

## Verification

### Host verification

- Build the default SDL2/H150101 target available in the repository.
- Verify that existing compilation warnings do not increase.
- Add focused tests where practical for platform-specific default selection and
  profile-counter reset behavior.

### Device verification

Run the same Sonic scene for at least one minute and confirm:

- `Frame rate: 60.000000` remains unchanged.
- Audio underruns stop or become rare isolated events.
- Core-run cadence stays close to 60 Hz.
- Skipped frames are isolated rather than consecutive.
- RGB565 update time decreases or its variance improves.
- Entering and leaving the menu preserves a valid background.
- HUD messages still render correctly.

If texture update remains near 7 to 8 ms after removing the redundant copy,
double-buffered streaming textures will be evaluated as a separate change.
